# Описание структуры базы данных KUDAB
## Схема `telegram` (2025)

Схема `telegram` хранит данные о пользователях Telegram, связанных с доменной таблицей `public.users`.

---

## 1. Список таблиц схемы `telegram`

- `telegram.users` — пользователи Telegram и привязка к доменным пользователям.

---

## 2. Таблица `telegram.users`

**Назначение:** учёт Telegram-пользователей, их базового профиля, привязки к доменному пользователю и предпочтений.

### Структура

| Поле             | Тип                                | Описание                                                                 |
|------------------|------------------------------------|--------------------------------------------------------------------------|
| `id`             | `bigint` (PK, default sequence)    | Уникальный идентификатор записи.                                        |
| `user_id`        | `bigint` (FK → `public.users.id`)  | ID доменного пользователя; `NULL`, если не привязан.                    |
| `telegram_id`    | `bigint` NOT NULL, UNIQUE          | Уникальный идентификатор пользователя Telegram (`from.id`).             |
| `telegram_username` | `varchar(255)`                  | `@username` (может отсутствовать/меняться).                              |
| `first_name`     | `varchar(255)`                     | Имя в Telegram.                                                          |
| `last_name`      | `varchar(255)`                     | Фамилия в Telegram.                                                      |
| `language_code`  | `varchar(8)`                       | Код языка, например `ru`, `en`, `en-US`.                                 |
| `chat_id`        | `bigint`                           | ID приватного чата с пользователем (`chat.id`).                          |
| `is_bot`         | `boolean` NOT NULL DEFAULT `false` | Флаг: является ли аккаунт ботом.                                        |
| `registered_at`  | `timestamp(0)`                     | Когда пользователь впервые зафиксирован в системе.                       |
| `last_active`    | `timestamp(0)`                     | Время последней активности пользователя.                                 |
| `created_at`     | `timestamp(0)`                     | Дата создания записи.                                                    |
| `updated_at`     | `timestamp(0)`                     | Дата последнего обновления записи.                                      |
| `city_id`        | `bigint`                           | ID города из доменного справочника; связь логическая, без FK.           |
| `prefs`          | `jsonb` NOT NULL DEFAULT `'{}'`    | JSON-настройки/предпочтения пользователя (фильтры, флаги и т.п.).       |

### Ограничения

- `PRIMARY KEY (id)`
- `UNIQUE (telegram_id)`  
  (в дампе два уникальных ограничения на `telegram_id`, фактически обеспечивают одно и то же условие).
- `FOREIGN KEY (user_id) REFERENCES public.users(id) ON DELETE SET NULL`  
  При удалении доменного пользователя связь обнуляется.

### Индексы

- `telegram_users_telegram_id_index` — BTree по `telegram_id`  
  Используется для быстрого поиска по Telegram ID.
- `telegram_users_user_id_index` — BTree по `user_id`  
  Для поиска по связанному доменному пользователю.
- `tg_users_city_id_idx` — BTree по `city_id`  
  Для выборок по городу (например, фильтрация рассылок по городу).

---

## 3. Особенности использования

- `telegram_id` — главный стабильный идентификатор в Telegram-слое; через него однозначно ищем пользователя.
- `user_id` позволяет связать Telegram-пользователя с доменным `public.users` и использовать общие ACL/роли.
- `prefs` в `jsonb` удобен для хранения произвольных флагов и настроек без миграций схемы.
- `city_id` даёт привязку к городу (по ID из доменного справочника), но FK нет — нужно учитывать это в коде (валидировать на уровне приложения/бота).
